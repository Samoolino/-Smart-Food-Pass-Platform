# VICTUALS Frontend Landing Page Functionality

This document defines the first visible frontend scope for VICTUALS.

## Frontend objective

Create an immersive, fast, transaction-ready e-commerce experience where users, merchants, subscribers, and administrators understand how VICTUALS works and can enter their correct app surface.

## Public landing pages

### 1. Home

Purpose: Establish the VICTUALS brand and platform value.

Required content:

- VICTUALS — Smart Nutritional Food Plan Pass.
- $WO-powered nutritional access.
- Multi-market merchant choice.
- Accredited merchants and product registry.
- Smart contract proof overview.
- West Africa pilot with global expansion readiness.

Primary actions:

- Access My Pass
- Start a Plan Package
- Join as Merchant
- Explore Market Stores
- Admin Console

### 2. How It Works

Purpose: Explain the operational flow.

Flow:

1. Subscriber funds plan/package.
2. Package value is represented as $WO.
3. User receives scheduled $WO pass credits.
4. Product and nutrition matrix determines eligible product scope.
5. User selects merchant market space.
6. Basket is suggested and priced.
7. QR redemption is generated.
8. Merchant validates and captures transaction.
9. Merchant receives $WO credit.
10. Merchant requests settlement conversion.

### 3. Market Stores

Purpose: Show merchant/e-commerce options.

Required content:

- Country and region selector.
- Merchant store cards.
- Store source type: API, inventory file, dashboard entry, registry selection.
- Eligible products count.
- Product category highlights.
- Delivery/pickup/physical redemption status where available.
- $WO acceptance status.

### 4. Plans and Packages

Purpose: Explain subscription options.

Required content:

- General nutritional access package.
- Family package.
- Target nutrition package.
- CSR/welfare package.
- Emergency relief package.
- Institution-backed package.
- Package value in local currency and $WO.
- Credit schedule and validity.
- Nutrition target and product scope.

### 5. Product and Nutritional Intelligence

Purpose: Explain how products are approved and suggested.

Required content:

- Product Registry.
- Nutrition Composition Registry.
- Nutritional Eligibility Matrix.
- Merchant price approval.
- Combined pass scope.
- Suggested basket preview.

### 6. Merchant Network

Purpose: Explain merchant onboarding and transactions.

Required content:

- Merchant accreditation.
- Inventory integration options.
- Price validation.
- QR scan and capture.
- $WO merchant balance.
- Settlement conversion.

### 7. Contract Overview

Purpose: Make contract functionality understandable to non-technical users.

Show these contract-backed modules as proof cards:

- Subscription Plan Registry
- Victuals Pass Manager
- Entitlement Scheduler
- Nutrition Policy Anchor
- Redemption Verifier
- Settlement Anchor

Each card should show:

- what it controls
- what proof it anchors
- app surface that uses it

## Authenticated access points

### User app

- Email/password login.
- Transaction accreditation status.
- Active pass set.
- Combined nutritional scope.
- $WO wallet view.
- Merchant market selection.
- Basket suggestion.
- QR generation.
- Redemption history.

### Merchant app

- Merchant accreditation status.
- Store/inventory source setup.
- Product price approval.
- QR scan.
- Transaction capture.
- $WO merchant balance.
- Settlement conversion request.

### Funder portal

- Create package.
- Select nutrition target.
- Fund package.
- Allocate users or groups.
- Track $WO release and usage.
- View impact reports.

### Admin console

- Treasury lower-limit and $WO release capacity.
- Plan/package approval.
- Product registry approval.
- Nutrition policy versioning.
- Merchant accreditation.
- Redemption monitoring.
- Settlement batch review.
- Network and contract address registry.

## UX behavior

- Always show $WO value.
- Show local currency equivalent when market is selected.
- Show country and region context.
- Show transaction readiness based on accreditation state.
- Show whether a merchant is API-connected, file-fed, manually managed, or registry-selected.
- Show product eligibility before QR generation.
